# 数据血缘分析平台建设需求说明书（完整版）

## 1. 文档说明

### 1.1 文档名称
数据血缘分析平台建设需求说明书

### 1.2 适用范围
本说明书适用于面向 Oracle、OceanBase、GBase、崖山、TDH 等异构数据库及大数据平台的血缘分析平台建设，适用于监管报送、数据仓库、数据集市、指标加工、批量任务排障、数据影响分析等场景。

### 1.3 编写目的
为解决监管报送系统、数据仓库系统等场景下结果数据问题定位困难、加工链路不透明、字段来源解释困难、问题排查耗时长等问题，建设统一的数据血缘分析平台，实现从贴源层到结果层的对象级、字段级、作业级、批次级血缘统一采集、统一建模、统一查询、统一展示。

### 1.4 读者对象
- 信息科技部架构管理人员
- 数据平台建设团队
- 数据仓库开发团队
- 运维与生产支持团队
- 监管报送开发与运维团队
- 数据治理团队
- 项目管理与采购管理人员

## 2. 建设背景与现状问题

### 2.1 建设背景
当前银行内部同时存在传统关系型数据库、国产数据库和大数据平台并存的技术格局，典型数据加工链路横跨 Oracle、OceanBase、GBase、崖山以及 TDH 等平台。监管报送、经营分析、客户画像、风险监测等结果数据往往经过多层数据加工后形成，涉及大量表、字段、SQL、存储过程、调度任务与批次运行。

随着系统数量增加、数据链路变长、加工逻辑复杂度提升，结果数据出现异常时，运维、开发和数据人员需要跨多个平台手工查找上游来源、加工逻辑和任务链路，排查过程耗时长、依赖经验强、结果不稳定，已成为数据系统稳定运行的重要痛点。

### 2.2 现状问题

#### 2.2.1 对象级加工链路不透明
- 结果表来自哪些上游表、经过哪些中间层表，缺乏统一视图。
- 监管报送结果表的完整加工链路通常散落在 SQL、调度配置、开发文档中，难以快速拼接。
- 多平台之间缺乏统一命名、统一血缘表达和统一查询入口。

#### 2.2.2 字段级来源解释困难
- 业务最关心的通常不是"表从哪里来"，而是"某个字段从哪里来、如何算出来"。
- 当前缺少字段级最小血缘链路，无法快速回答某个指标字段、报送字段、标签字段的来源与计算路径。
- 即使部分系统能够展示全量上下游图，也往往过大、过杂，不利于排障。

#### 2.2.3 异构平台采集难度高
- Oracle、OceanBase、GBase、崖山等关系型数据库以 SQL、存储过程为主。
- TDH 场景下存在 Inceptor、Spark、Hive 类任务、调度依赖和运行时链路。
- 不同平台 SQL 方言、元数据接口、作业机制存在差异，缺乏统一采集框架。

#### 2.2.4 运行态与设计态脱节
- 文档中描述的加工逻辑与实际生产运行可能不一致。
- 批次失败、表结构变更、任务改造后，既有文档和人工台账容易失效。
- 缺乏对真实运行链路、最近变更和最近批次状态的关联展示。

#### 2.2.5 排障效率低
- 遇到数据问题时，通常需要 DBA、开发、运维多角色协作排查。
- 问题排查严重依赖资深人员经验。
- 难以快速回答"问题字段受哪些源字段影响""改动某字段会影响哪些下游结果"。

### 2.3 建设必要性
建设统一血缘分析平台的必要性体现在：
- 支撑监管报送口径可解释与审计追溯。
- 提升数据仓库及报送系统问题定位效率。
- 支撑系统改造、字段调整、口径变更的影响分析。
- 形成统一元数据和血缘资产底座，为后续数据治理、质量管理、口径管理奠定基础。

## 3. 建设目标与建设原则

### 3.1 总体目标
建设统一的数据血缘分析平台，实现对 Oracle、OceanBase、GBase、崖山、TDH 等平台的表级、字段级、作业级、批次级血缘统一采集、统一建模、统一管理和统一展示，重点解决"结果表完整加工血缘可查"和"字段最小血缘链路可追"的核心问题。

### 3.2 分阶段目标

#### 3.2.1 一期目标
- 建立统一平台与统一血缘模型。
- 优先接入 Oracle、TDH、崖山。
- 实现结果表完整对象级血缘查询。
- 实现关键字段最小血缘链路查询。
- 支持问题排查页，关联最近批次和运行状态。

#### 3.2.2 二期目标
- 扩展接入 OceanBase 和 GBase。
- 补齐存储过程、动态 SQL、复杂表达式解析能力。
- 增强影响分析与变更审计能力。
- 支持人工修正和审核确认流程。

#### 3.2.3 三期目标
- 与数据质量平台联动。
- 与调度平台、开发平台、元数据治理平台集成。
- 支持统一指标口径解释、监管问询快速响应和数据问题智能排查。

### 3.3 建设原则

#### 3.3.1 统一建模原则
对不同数据库和计算引擎的元数据、血缘关系、作业关系进行统一抽象，形成一致的数据资产与血缘模型。

#### 3.3.2 分层采集原则
关系型数据库以元数据采集与 SQL 静态解析为主；TDH 以运行时采集与 SQL 解析结合为主；人工补录作为兜底手段。

#### 3.3.3 自动优先、人工兜底原则
优先依赖自动采集和自动解析获得血缘；对复杂存储过程、动态 SQL、非标准加工链路提供人工修正和补录机制。

#### 3.3.4 最小可用原则
优先围绕"完整血缘"和"最小链路"两类最有价值能力落地，避免平台建设初期过度追求大而全。

#### 3.3.5 可审计原则
自动解析结果、人工修正结果、审核确认结果都必须保留时间、人员、来源和版本痕迹，满足审计要求。

#### 3.3.6 可扩展原则
平台应支持后续平滑接入更多数据库、数据开发工具、调度平台和治理能力。

## 4. 建设范围

### 4.1 技术范围
纳入建设范围的数据平台包括但不限于：
- Oracle

- GBase（需区分不同产品线，如 8a、8s）

### 4.2 业务范围
首期建议覆盖以下高价值场景：
- 监管报送结果表及关键字段
- 数据仓库核心主题域结果表
- 经营管理驾驶舱关键指标
- 风险、合规、财务等重点报表

### 4.3 能力范围
平台建设范围包括：
- 元数据采集
- SQL 解析
- 运行时血缘采集
- 血缘统一建模
- 对象级完整血缘查询
- 字段级最小血缘链路查询
- 影响分析
- 运行态关联分析
- 人工修正与审核
- 血缘图可视化展示
- 对外 API 服务

### 4.4 不在本期范围内的内容
- 不负责替代现有调度平台、开发平台、数据质量平台。

- 不在本期建设中替代现有 CMDB、主数据、数据标准主平台。

## 5. 总体架构设计

### 5.1 总体架构目标
构建"采集层—解析层—建模层—存储层—服务层—展示层"六层架构，支撑多源异构血缘的统一处理。

### 5.2 架构分层

#### 5.2.1 数据源层
包括：
- Oracle 数据库
- OceanBase 数据库
- GBase 数据库
- 崖山数据库
- TDH Inceptor / Spark / Hive 类平台
- 调度平台
- 开发文档、人工补录信息

#### 5.2.2 采集层
负责采集各类元数据和血缘原始信息，包括：
- JDBC 元数据采集器
- 数据库系统表采集器
- SQL 脚本采集器
- 存储过程源码采集器
- TDH 任务与运行信息采集器
- 调度依赖采集器
- 人工维护采集器

#### 5.2.3 解析层
负责对原始采集数据进行标准化和血缘提取，包括：
- SQL 方言识别模块
- SQL 预处理模块
- SQL 解析引擎
- 表级血缘提取模块
- 字段级血缘提取模块
- 表达式与规则抽取模块
- 运行时血缘转换模块

#### 5.2.4 建模层
负责将各类解析结果统一建模，包括：
- 数据源模型
- 库/模式模型
- 表模型
- 字段模型
- 作业模型
- SQL 语句模型
- 批次运行模型
- 任务依赖模型
- 血缘边模型
- 审核与修正模型

#### 5.2.5 存储层
建议采用双存储模式：
- 元数据与资产管理存储：元数据平台/关系库存储
- 图查询与最短路径计算存储：图数据库存储

#### 5.2.6 服务层
提供统一业务服务，包括：
- 元数据服务
- 血缘查询服务
- 最小链路计算服务
- 影响分析服务
- 运行态关联服务
- 人工修正服务
- 审核服务
- 对外开放 API

#### 5.2.7 展示层
提供统一可视化页面和查询入口，包括：
- 资产搜索页
- 完整血缘页
- 字段最小链路页
- 影响分析页
- 问题排查页
- 人工修正页
- 采集监控页
- 权限与系统管理页

## 6. 建设模式与开源技术路线

### 6.1 总体技术路线
采用"开源平台底座 + 自定义采集适配器 + SQL 解析增强 + 图查询增强"的建设模式。

### 6.2 推荐建设方式
建议以元数据平台作为统一底座，以自研适配器解决国产数据库和异构平台接入问题，以图查询能力解决字段最小链路与路径裁剪问题。

### 6.3 技术选型评估

#### 6.3.1 元数据平台选型对比

| 评估维度 | DataHub | Apache Atlas | OpenMetadata | 权重 |
|---------|---------|--------------|--------------|------|
| 社区活跃度（GitHub Stars） | 18k+ | 8k+ | 7k+ | 15% |
| 血缘可视化能力 | 优秀 | 一般 | 良好 | 25% |
| 自定义扩展性 | 支持 Plugin | 需二次开发 | 支持 Plugin | 20% |
| 中文文档完善度 | 部分 | 较少 | 良好 | 10% |
| 与图数据库集成 | 原生支持 Neo4j | 需定制 | 支持 JanusGraph | 20% |
| 企业案例（金融行业） | 有（某股份制银行） | 有（某国有银行） | 较少 | 10% |

**推荐方案**：DataHub + Neo4j  
**备选方案**：OpenMetadata + JanusGraph（如 DataHub 许可证受限）

#### 6.3.2 SQL 解析引擎选型

| 引擎 | 支持方言 | 可扩展性 | 中文社区 | 推荐场景 |
|------|---------|---------|---------|---------|
| JSqlParser | Oracle/MySQL/PostgreSQL | 支持自定义语法 | 活跃 | Oracle/OceanBase |
| Druid SQL Parser | 支持国产数据库 | 阿里巴巴维护 | 活跃 | GBase/崖山 |
| ANTLR + 自定义语法 | 完全自定义 | 高 | 需自建团队 | TDH Inceptor |

**建议**：采用多引擎策略，按数据库类型路由到对应解析器

### 6.4 自定义建设重点
由于国产数据库和异构平台接入能力差异较大，必须重点建设以下自定义能力：
- OceanBase 模式识别与适配
- GBase 8a / 8s 分产品线适配
- 崖山 SQL 与存储过程解析增强
- TDH Inceptor / Spark 血缘采集适配
- 存储过程、动态 SQL、复杂脚本解析能力
- 字段最小链路算法与图查询服务

## 7. 业务需求

### 7.1 核心业务需求
平台需满足以下核心业务需求：
- 快速查询结果表从贴源层到结果层的完整加工链路。
- 快速查询结果字段的最小血缘链路。
- 支持监管报送字段口径解释与追溯。
- 支持问题字段的上游排查、下游影响分析。
- 支持开发、运维、数据治理等多角色协同使用。

### 7.2 典型业务场景

#### 7.2.1 监管报送问题排查
当监管报送某字段异常时，运维人员可输入结果表和字段，快速查看字段来源、加工步骤、上游表字段、最近运行批次及相关作业信息。

#### 7.2.2 数据仓库指标核查
当经营指标口径争议或数据差异发生时，数据开发人员可查询指标字段的最小血缘链路，确认具体来源字段、聚合逻辑和过滤条件。

#### 7.2.3 系统改造影响分析
当上游表字段调整或某个加工逻辑变更时，架构管理人员和数据开发人员可评估受影响的下游表、字段、报表和监管报送项。

#### 7.2.4 审计追溯
当审计人员要求说明某数据项来源和加工过程时，可导出完整加工血缘和字段最小链路作为支撑材料。

### 7.3 用户故事与验收标准

#### 7.3.1 运维人员排障场景
**用户故事**：
> 作为运维人员张工（5 年银行系统运维经验），  
> 当监管报送系统出现数据异常时，  
> 我希望在 3 分钟内定位到问题字段的上游来源和加工链路，  
> 以便快速修复并恢复报送。

**前置条件**：
- 问题表已完成血缘采集
- 最近 24 小时内有批次运行记录

**验收标准**：
- 输入表名 + 字段名后，10 秒内展示完整血缘链路
- 自动高亮显示最近失败的任务节点
- 可一键导出包含 SQL、任务配置、运行日志的排查报告

**异常场景**：
- 血缘解析不完整时，系统应提示"该字段血缘完整度 75%，建议人工补录"
- 数据源不可达时，应显示"最近采集时间：2024-01-15 03:00，采集状态：失败"

#### 7.3.2 架构师影响分析场景
**用户故事**：
> 作为架构师李工，  
> 当需要调整上游表 T1 的字段结构时，  
> 我希望准确评估会影响哪些下游报表和监管报送项，  
> 以便提前通知相关团队并制定回退方案。

**验收标准**：
- 选择表 T1 后，30 秒内列出所有直接/间接受影响的下游对象
- 影响路径应标注层级深度（如：T1 → T2 → T3，深度=2）
- 对监管报送表应使用特殊标识（如红色边框）

## 8.1 资产搜索与定位模块

### 8.1.1 功能目标
提供统一搜索入口，支持按系统、数据库、schema、表、字段、任务、结果表等多种对象快速定位。

### 8.1.2 功能要求
- 支持按表名精确查询与模糊查询。
- 支持按字段名精确查询与模糊查询。
- 支持按系统名、库名、schema 查询。
- 支持按业务域、主题域、监管场景过滤。
- 支持按对象类型筛选，如表、字段、任务、批次。
- 搜索结果需展示基础信息、所属系统、最近更新时间、是否存在血缘、是否有人工修正。

### 8.1.3 页面要素
- 搜索框
- 高级筛选区
- 搜索结果列表
- 结果快捷跳转按钮

## 8.2 对象级完整血缘模块

### 8.2.1 功能目标
针对某个结果表、视图、结果数据集，展示从贴源层到结果层的完整加工链路。

### 8.2.2 功能要求
- 输入结果表后，展示其所有直接和间接上游对象。
- 支持展示下游对象，用于影响分析。
- 支持展示中间加工层，如 ODS、DWD、DWS、DM、应用结果层。
- 支持展示任务节点、SQL 节点、系统节点。
- 支持按照层级展开与收缩。
- 支持切换"仅表视图""表 + 任务视图""全量视图"。
- 支持限定查询深度。
- 支持筛选仅展示某个系统、某个主题域、某个时间范围内链路。

### 8.2.3 展示要求
- 图形展示应支持节点拖拽、缩放、全屏、导出图片。
- 节点点击后应展示详细属性，包括系统、schema、owner、最近采集时间、最近运行批次、备注说明。
- 不同节点类型应有明显区分，如表、字段、任务、SQL、人工补录节点。

### 8.2.4 输出要求
- 支持导出对象级完整血缘图。
- 支持导出链路明细列表。
- 支持导出关联 SQL 与任务清单。

## 8.3 字段级最小血缘链路模块

## 9. 各数据源适配需求

## 9.1 Oracle 适配需求
- 支持基于 JDBC 与系统视图采集库、表、字段、索引、注释等元数据。
- 支持采集视图定义。
- 支持采集存储过程、函数、包体源码。
- 支持解析常见 Oracle SQL 与 PL/SQL。
- 支持与调度系统关联。

## 9.2 OceanBase 适配需求
- 支持识别 MySQL 模式与 Oracle 模式。
- 支持不同模式下的元数据采集。
- 支持按模式使用不同 SQL 解析规则。
- 支持视图、存储过程、函数等对象采集。

## 9.3 GBase 适配需求
- 需区分 GBase 8a 与 GBase 8s 等产品线。
- 支持不同产品线的元数据采集方式。
- 支持不同产品线的 SQL 方言适配。
- 支持基础对象、视图、脚本加工链路采集。

## 9.4 崖山适配需求
- 支持 JDBC 元数据采集。
- 支持 SQL 与存储过程解析。
- 支持视图、函数、包体等对象采集。
- 支持运行日志辅助解析与问题定位。

## 9.5 TDH 适配需求
- 不按单一数据库对象接入，应按组件拆解接入。
- 支持 Inceptor SQL 采集与解析。
- 支持 Spark 作业运行信息采集。
- 支持 Hive 类元数据采集。
- 支持调度依赖采集。
- 支持将运行时血缘与静态解析结果融合。

## 9.6 SQL 解析复杂度分级与应对策略

### 9.6.1 复杂度分级

| 等级 | 特征 | 占比预估 | 解析策略 | 目标准确率 |
|------|------|---------|---------|-----------|
| L1-简单 | 单表查询、简单 JOIN | 40% | 静态解析 | ≥98% |
| L2-中等 | 多层嵌套、子查询、CTE | 35% | 静态解析 + 语义分析 | ≥90% |
| L3-复杂 | 存储过程、游标、临时表 | 20% | 静态解析 + 人工补录 | ≥80% |
| L4-极复杂 | 动态 SQL、反射调用、跨库链接 | 5% | 运行时采集 + 人工补录 | ≥70% |

### 9.6.2 解析失败兜底机制

**触发条件**：
- 解析器抛出 SyntaxError 或 TimeoutException
- 解析结果可信度 < 60%

**兜底流程**：
1. 系统自动标记为"解析失败，待人工补录"
2. 推送工单给对应数据源负责人
3. 人工补录后进入审核流程
4. 积累为回归测试用例

### 9.6.3 存储过程解析策略

**不支持的场景**（需人工补录）：
- 使用 EXECUTE IMMEDIATE 的动态 SQL
- 通过 DBMS_SQL 包执行的动态语句
- 跨数据库链接（Database Link）的间接依赖
- 条件分支中动态创建的临时表

**支持的场景**：
- 静态 INSERT/UPDATE/DELETE/SELECT
- 静态游标操作
- 确定的函数调用链

### 10.1 核心实体模型
平台至少应包含以下实体：

- 表

- 血缘边
- 人工修正记录
- 审核记录

### 10.2 血缘关系模型
需支持以下关系类型：
- 表到表血缘
- 字段到字段血缘
- 作业读取表关系
- 作业写入表关系
- SQL 生成关系
- 批次运行关系
- 人工修正关系
- 审核确认关系

### 10.3 血缘边属性要求
每条血缘边应至少包含：

**基础属性**：
- edge_id: 全局唯一标识（UUID）
- edge_type: 表级血缘/字段级血缘/作业依赖/批次依赖
- source_node: 来源节点 ID
- target_node: 目标节点 ID
- source_type: TABLE/FIELD/JOB/BATCH

**来源属性**：
- derivation_method: STATIC_PARSE/RUNTIME_CAPTURE/MANUAL_ENTRY/MANUAL_REVIEW
- parse_engine: JSqlParser/Druid/ANTLR/RUNTIME_LOG
- confidence_score: 0.0-1.0 可信度评分
- evidence: 解析依据（SQL 片段/日志片段/人工说明）

**时间属性**：
- created_at: 创建时间
- updated_at: 更新时间
- valid_from: 生效时间（用于处理历史版本）
- valid_to: 失效时间（用于处理结构变更）

**审计属性**：
- created_by: 创建人/创建任务
- reviewed_by: 审核人
- review_comment: 审核意见
- version: 版本号（乐观锁）

**业务属性**：
- transformation_type: DIRECT_MAP/CALCULATION/AGGREGATION/FILTER/JOIN
- expression: 转换表达式（如：`SUM(amount) AS total_amount`）
- business_term: 关联的业务术语（如："存款余额"）
- regulatory_tag: 监管标识（如："EAST 5.0-表 3-字段 12"）

### 10.4 版本管理模型

#### 10.4.1 表结构变更版本管理

**场景**：当上游表 T1 增加字段或修改字段类型时

**模型设计**：
```
Table_Version {
  table_id
  version_number: v1.0, v2.0, ...
  schema_snapshot: JSON 格式的字段定义
  valid_from: 生效时间
  valid_to: 失效时间（NULL 表示当前版本）
  change_reason: 变更原因
  change_request_id: 关联的变更工单
}
```

**血缘版本关联**：
- 每条血缘边关联 source_table_version 和 target_table_version
- 查询时默认使用最新版本，支持历史时点查询

#### 10.4.2 血缘快照管理

**需求**：满足审计追溯要求，可重现历史时点的血缘关系

**实现**：
- 每日凌晨对全量血缘图生成快照
- 重大变更（如核心表结构调整）时生成临时快照
- 快照保留策略：最近 3 年全量，3 年前按季度归档

## 11. 流程需求

## 11.1 元数据采集流程
1. 配置数据源连接信息。
2. 定时或手工触发采集任务。
3. 采集表、字段、视图、过程等元数据。
4. 进行标准化和建模。
5. 入库并生成采集报告。

## 11.2 SQL 血缘解析流程
1. 获取 SQL 脚本、视图定义、存储过程源码。
2. 进行 SQL 方言识别和预处理。
3. 进行 AST 解析。
4. 提取表级血缘和字段级血缘。
5. 生成表达式和规则信息。
6. 入图并等待展示。

## 11.3 运行时血缘采集流程
1. 采集作业启动、读取、写入、结束等事件。
2. 提取作业与数据集关系。
3. 关联批次、任务、表对象。
4. 与静态血缘进行融合。
5. 生成运行态血缘结果。

## 11.4 问题排查流程
1. 运维人员输入问题结果表或字段。
2. 系统展示完整血缘和最小链路。
3. 系统联动展示最近批次、失败作业、最近变更。
4. 用户定位问题上游节点。
5. 如自动结果不完整，可发起人工修正。

## 11.5 人工修正与审核流程
1. 用户提交血缘修正申请。
2. 系统记录修正内容与理由。
3. 审核人员进行审核。
4. 审核通过后写入正式血缘图。
5. 全流程保留审计记录。

## 17. 风险与控制措施

### 17.1 主要风险
- 异构数据库方言复杂，自动解析准确率不足。
- TDH 运行链路接入复杂，运行时采集难度较大。
- 存储过程、动态 SQL、非标准脚本加工难以全自动识别。
- 平台建设初期对象量大，查询性能可能受到影响。

### 17.2 控制措施
- 优先聚焦高价值场景，逐步扩展。
- 建立人工修正与审核机制。
- 建立方言规则库和解析回归测试集。
- 建立分层缓存与图查询优化机制。
- 建立采集、解析、展示全链路监控机制。

## 20. 验收测试标准

### 20.1 功能验收测试

#### 20.1.1 表级血缘查询测试

**测试用例 TC-001：简单单表血缘查询**
- **前置条件**：表 T1 的血缘已采集完成
- **测试步骤**：
  1. 搜索框输入表名 T1
  2. 点击"查看完整血缘"
- **预期结果**：
  - 展示 T1 的所有上游表（至少 3 层）
  - 展示所有下游表（如有）
  - 响应时间≤3 秒
- **通过标准**：血缘图与人工核对一致

**测试用例 TC-002：跨系统血缘查询**
- **前置条件**：表 T1 涉及 Oracle→TDH→GBase 跨系统加工
- **测试步骤**：
  1. 搜索表 T1
  2. 切换"仅表视图"→"表 + 任务视图"
- **预期结果**：
  - 正确展示跨系统链路
  - 系统节点用不同颜色区分
- **通过标准**：跨系统链路完整度≥90%

#### 20.1.2 字段级最小链路测试

**测试用例 TC-010：简单映射字段**
- **前置条件**：字段 F1 为直接映射（SELECT F1 AS F2）
- **测试步骤**：
  1. 输入目标表 T2 和字段 F2
  2. 点击"查看最小链路"
- **预期结果**：
  - 仅展示 F2→F1 的直接路径
  - 不展示无关分支
- **通过标准**：路径节点数=2

**测试用例 TC-011：聚合计算字段**
- **前置条件**：字段 F3 = SUM(F1) + AVG(F2)
- **测试步骤**：
  1. 输入目标字段 F3
  2. 查看血缘链路
- **预期结果**：
  - 展示 F3←F1 和 F3←F2 两条路径
  - 展示聚合表达式 `SUM(F1) + AVG(F2)`
- **通过标准**：表达式解析正确

### 20.2 性能验收测试

#### 20.2.1 压力测试

**测试场景**：50 并发用户同时查询血缘
- **测试工具**：JMeter
- **持续时间**：30 分钟
- **通过标准**：
  - P95 响应时间≤5 秒
  - 错误率≤0.1%
  - 系统无崩溃、无内存泄漏

### 20.3 准确率验收测试

#### 20.3.1 解析准确率测试

**测试集**：
- 从生产环境抽取 1000 个典型 SQL 脚本
- 人工标注标准血缘关系作为 Ground Truth
- 覆盖 L1-L4 所有复杂度等级

**通过标准**：
- L1 简单场景：准确率≥98%
- L2 中等场景：准确率≥90%
- L3 复杂场景：准确率≥80%
- L4 极复杂场景：准确率≥70%
