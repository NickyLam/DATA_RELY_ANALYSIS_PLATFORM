# 2026-05-11 工作日志

## 修复：字段血缘查询 source_table 为空导致无结果

### 问题描述
查询 `RRP_MDL.O_ICL_CMM_INDV_CUST_BASIC_INFO.CUST_NAME` 时没有显示血缘关系，但实际该字段由存储过程 `ETL_O_ICL_CMM_INDV_CUST_BASIC_INFO` 从 `ICL.V_CMM_INDV_CUST_BASIC_INFO`（外部视图）直传而来。

### 根因
存储过程来源表 `ICL.V_CMM_INDV_CUST_BASIC_INFO` 是 ICL schema 的外部视图，不在 .tab 文件中。当 INSERT INTO ... SELECT 语句的 SELECT 列不带表前缀（如 `CUST_NAME` 而非 `A.CUST_NAME`），且 FROM 子句只有一张表时，解析器 `EnhancedProcedureParser._extract_insert_mappings` 无法将裸列名关联到来源表，导致 field_mapping 的 `source_table` 设为空字符串。tracer 在查找时跳过 `source_table` 为空的记录。

### 修复（两处）

1. **`core/procedure_parser.py` — _extract_insert_mappings**：
   - 新增从 FROM 子句提取第一个裸表名的逻辑（`from_default_table`）
   - 当 SELECT 列无表前缀且 `source_table` 为空时，用 `from_default_table` 自动补全
   - 影响范围：抽样 20 个过程修复后残留率 1.2%（多表/子查询场景）

2. **`core/lineage_tracer.py` — _find_source_fields + _find_source_fields_in_procedure**：
   - 新增 `_infer_source_table_from_lineage()` 方法
   - 当 field_mapping 的 `source_table` 为空但 `source_column` 非空时，从同过程的 table_lineages 推断来源表
   - 优先匹配同过程，无匹配时若候选唯一则使用
   - 作为解析器修复的兜底，处理已有的空 source_table 数据

### 验证结果
- 修复前：99 个 field_mappings 全部 source_table 为空，查询返回 0 条链路
- 修复后：99 个 field_mappings source_table 正确填充为 `ICL.V_CMM_INDV_CUST_BASIC_INFO`，查询返回 1 条链路（2 个节点、1 条边）

## 新增：字段级下游追溯功能

### 问题描述
API 的 `mode=downstream` 之前未实现，直接回退到上游追溯，导致上下游查询返回相同结果。

### 实现
1. **`core/lineage_tracer.py`**：
   - 新增 `_source_field_mapping_idx` 反向索引（source_table → source_field → [FieldMapping]），在 `_build_index` 中构建
   - 新增 `_TargetRecord` 数据类（与 `_SourceRecord` 对称）
   - 新增 `trace_field_downstream()` 公开方法
   - 新增 `_bfs_trace_downstream()` 方法（BFS 向下扩展，从起始字段查找所有下游目标字段）
   - 新增 `_find_target_fields()` 方法（通过反向索引查找下游目标）
   - 新增 `_fuzzy_match_source_table_key()` 方法（模糊匹配 schema 前缀差异）

2. **`api_server.py` — `_handle_lineage_query`**：
   - `mode == "downstream"` 分支现在调用 `tracer.trace_field_downstream()`
   - 返回值包装为 `SimpleNamespace`（模拟 FieldLineageResult），供 `_result_to_graph` 使用

### 验证结果
- `CUST_NAME` 上游：2 节点、1 边（ICL.V_CMM_INDV_CUST_BASIC_INFO → O_ICL_CMM_INDV_CUST_BASIC_INFO）

## 修复：跨 schema 表名模糊匹配

### 问题描述
查询 `RRP_MDL.EAST5_201_GRJCXXB.KHXM` 无任何血缘结果。该表同时存在于 `RRP_MDL` 和 `RRP_EAST` 两个 schema，但存储过程 `ETL_EAST5_201_GRJCXXB` 的 target_table 是 `RRP_EAST.EAST5_201_GRJCXXB`。

### 根因
`_fuzzy_match_table_key` 和 `_fuzzy_match_source_table_key` 的匹配逻辑仅支持"后缀匹配"（`key.endswith(table_name)`），不支持"裸表名跨 schema 匹配"。当用户输入 `RRP_MDL.EAST5_201_GRJCXXB` 时，无法匹配到索引中的 `RRP_EAST.EAST5_201_GRJCXXB`。

### 修复
两个模糊匹配方法均增加"裸表名匹配"策略：
- 去掉双方 schema 前缀后比较表名部分
- 例如 `RRP_MDL.EAST5_201_GRJCXXB` → 裸表名 `EAST5_201_GRJCXXB` → 匹配 `RRP_EAST.EAST5_201_GRJCXXB`

### 验证
- `RRP_MDL.EAST5_201_GRJCXXB.KHXM` 上游：5 节点、5 边（ICL.V_CMM_INDV_CUST_BASIC_INFO → ... → EAST5_201_GRJCXXB）
- `O_ICL_CMM_INDV_CUST_BASIC_INFO.CUST_NAME` 下游：49 节点、49 边（未受影响）

## 修复：下游追溯 schema 变体合并 + 环路检测 + 链路方向

### 问题描述
查询 `O_ICL_CMM_INDV_CUST_BASIC_INFO.CUST_NAME` 的下游只返回 1 条结果（`M_EAST_DERIV_CPTL_DTL.BUYER_NM`），但 `RRP_MDL.O_ICL_CMM_INDV_CUST_BASIC_INFO.CUST_NAME` 有 21 条下游映射。裸表名查询遗漏了带 schema 前缀变体的数据。

### 根因（三个问题）
1. **schema 变体未合并**：`_find_target_fields` 查到裸表名 key 后直接返回，不收集 `RRP_MDL.O_ICL_CMM_INDV_CUST_BASIC_INFO` 等同裸表名的其他变体数据。`_find_source_fields` 同样存在此问题。
2. **BFS visited 不检测 schema 变体**：`O_ICL_CMM_INDV_CUST_BASIC_INFO.CUST_NAME` 和 `RRP_MDL.O_ICL_CMM_INDV_CUST_BASIC_INFO.CUST_NAME` 是不同的 visited key，导致 schema 变体合并后中间节点又回到起点，形成 49 条"反向边"环路。
3. **下游链路方向反转**：`_build_chains_from_bfs_tree` 从叶子→根构建链路（leaf→root），对上游追溯正确（源头→目标），但对下游追溯方向反了（终点→起点而非起点→终点）。

### 修复（`core/lineage_tracer.py` 三处）

1. **`_find_target_fields` 和 `_find_source_fields` 增加 schema 变体合并**：
   - 查询裸表名时，遍历索引中所有 `SCHEMA.bare_name` 形式的 key，合并结果
   - 查询带 schema 的表名时，同时收集裸表名的结果
   - 使用 `seen_keys` 去重（`(target_table, target_field)` 或 `(source_table, source_field)`）

2. **BFS 循环检测增加裸表名 key**：
   - `_bfs_trace` 和 `_bfs_trace_downstream` 中，visited 集合同时注册完整 key 和裸表名 key
   - 防止 schema 变体导致的环路

3. **`_build_chains_from_bfs_tree` 增加 `reverse` 参数**：
   - 下游追溯时传入 `reverse=True`，将 leaf→root 链路反转为 起点→终点
   - 上游追溯默认 `reverse=False`（保持原有行为）

### 验证结果
- `O_ICL_CMM_INDV_CUST_BASIC_INFO.CUST_NAME` 下游：**51 节点、51 边、59 field_mappings、49 条链路**（修复前：2 节点、1 边）
- 反向边数量：**0**（修复前：49）
- 链路方向：`O_ICL → M_CUST_IND_INFO → M_CUST_IND_INFO_EAST → EAST5_*` ✓

## 修复：字段名大小写不敏感问题

### 问题描述
查询 `RRP_MDL.O_ICL_CMM_INDV_CUST_BASIC_INFO.cust_name`（小写 field）下游返回 0 结果，但大写 `CUST_NAME` 返回 55 节点、55 边。

### 根因
`trace_field_downstream` 和 `trace_field` 调用 `_build_chains_from_bfs_tree` 时，传入的 `(source_table, source_field)` 元组是**原始未 normalize 的值**。BFS 树中的 key 已被 normalize 为大写（如 `RRP_MDL.O_ICL_CMM_INDV_CUST_BASIC_INFO.CUST_NAME`），但 `_build_chains_from_bfs_tree` 用 `f"{target[0]}.{target[1]}"` 拼接 root_key 时用的是原始小写值（如 `RRP_MDL.O_ICL_CMM_INDV_CUST_BASIC_INFO.cust_name`），导致 key 不匹配，直接返回空链路。

### 修复（`core/lineage_tracer.py` 两处）
1. **`trace_field`**（第 188 行）：调用 `_bfs_trace` 和 `_build_chains_from_bfs_tree` 前先 normalize table 和 field
2. **`trace_field_downstream`**（第 433 行）：同上

### 验证结果
- 小写 `cust_name` 下游：55 节点、55 边（修复前：0）
- 大写 `CUST_NAME` 下游：55 节点、55 边（一致）
- 裸表名 + 小写 `O_ICL_CMM_INDV_CUST_BASIC_INFO.cust_name`：65 chains（修复后一致）
