## 任务：修复 Batch C 口径提取功能"死代码"问题

### 背景
Batch C（CTE/自定义函数/完整表达式提取）的代码虽然已实现，但实际运行时跟改造前无区别。深度调查发现三个根因导致 Batch C 代码是"死代码"：
1. `procedure_parser.py` 正则从 `INSERT`/`MERGE`/`UPDATE` 开始匹配，不包含前面的 `WITH`/`CTE` 子句
2. `api_server.py` 的 `_chains_to_caliber_result` else 分支直接创建裸 `CaliberInfo`，绕过了 `build_caliber_info`
3. `sql_block=""` 时所有提取器返回空值

### 修复内容

#### 修复1：`core/procedure_parser.py` — sql_block 包含 WITH/CTE 子句
- 新增 `_extend_with_cte()` 方法：在 DML 匹配后向前查找 `WITH` 关键字，如果找到且中间没有其他 DML 语句，则将 CTE 前缀拼接到 `sql_block`
- 修改 `_extract_all_sql_operations()`：对 INSERT/MERGE/UPDATE 三种操作类型都调用 `_extend_with_cte()`
- 逻辑安全保障：
  - 没有 WITH 的 SQL 不受影响（返回原始 raw_block）
  - WITH 和 INSERT 之间有其他 DML 时不扩展（避免错误关联）
  - WITH 前面紧邻字母时不匹配（排除子查询内部的 WITH）

#### 修复2：`api_server.py` — else 分支调用 `build_caliber_info`
- 修改 `_chains_to_caliber_result()` 的 else 分支：
  - 旧：直接创建 `CaliberInfo(...)` 裸对象，where_conditions=0, join_conditions=0, operation_type=""
  - 新：构造 `fallback_fm = FieldMapping(...)` 传给 `CaliberExtractor.build_caliber_info()`，确保 sql_block 中的 WHERE/JOIN 等信息被提取
  - confidence 设为 0.4（表示这是模糊匹配的口径信息）

### 验证结果

#### 单元测试（全部通过）
1. Test1：无 CTE 的 INSERT — sql_block 不变 ✅
2. Test2：WITH...INSERT — sql_block 正确扩展包含 CTE ✅
3. Test3：中间有其他 DML — 不误扩展 ✅
4. Test4：第一个 INSERT 的 WITH — 正确扩展 ✅

#### 端到端验证（POST /api/caliber/query）
查询 `RRP_MDL.EAST5_201_GRJCXXB.KHXM`：

| 步骤 | 修复前 | 修复后 |
|------|--------|--------|
| Step 1 (CUST_NM→CUST_NM_DESEN) | WHERE=0, JOIN=0, op_type="" | **WHERE=1, JOIN=3, op_type=INSERT_VALUES** |
| Step 2 (CUST_NM→CUST_NM) | WHERE=0, JOIN=0, op_type="" | **WHERE=1, JOIN=3, op_type=INSERT_VALUES** |
| Step 3 (CUST_NAME→CUST_NM) | WHERE=0, JOIN=0, op_type="" | **WHERE=1, JOIN=0, op_type=INSERT_SELECT** |

### 遗留问题
- Step 4/5 的 `op_type=""` 和 `WHERE=0` 是因为 `_extract_sql_block_for_mapping` 找不到对应的 SQLOperation（存储过程不在索引中或 target_table 不匹配），需要后续增强
- `_WHERE_PATTERN` 未能区分 CTE 内部 WHERE 和外层 WHERE，导致 CTE 场景下 WHERE 条件可能包含 CTE 内部子句
- 实际 .prc 文件中 WITH...INSERT 模式较少（多见 WITH...SELECT INTO），CTE 扩展修复的直接影响面有限，但 else 分支修复影响面很大

## 任务：修复两个 Bug — 表搜索 schema 匹配错误 + Pipeline 可视化布局

### Bug1：表搜索自动匹配到错误 schema

#### 根因
1. **后端 `_handle_list_tables`** 用 `short_name` 去重（`seen.add(name.upper())`），导致 `RRP_EAST.EAST5_201_GRJCXXB` 和 `RRP_MDL.EAST5_201_GRJCXXB` 只返回第一个遇到的
2. **排序仅按 `short_name`**，不考虑 schema 与搜索关键字的匹配度，`RRP_MDL` 排在前面
3. **前端搜索结果不显示 schema**，用户无法区分同名表的不同 schema
4. **前端自动补全**（`executeCaliberQuery`/`search-panel.js`）找到第一个 `short_name` 匹配就 break

#### 修复内容
1. **后端**：`seen` 改用 `full_name` 去重，允许不同 schema 同名表共存；增加 `schema` 字段；排序改为按匹配度（精确匹配→schema匹配→前缀匹配→短名→全名）
2. **前端 `caliber-tab.js`**：搜索结果项显示 `schema.table` 格式，自动补全逻辑改为智能匹配 schema（去数字后比对前缀重叠）
3. **前端 `search-panel.js`**：同步修复搜索结果显示 schema + 自动补全逻辑

#### 验证
搜索 `EAST5_201_GRJCXXB` → `RRP_EAST.EAST5_201_GRJCXXB` 排在第一位 ✅

### Bug2：Pipeline 可视化不符合"概览卡-链路图-详情面板"设计

#### 根因
1. **`.caliber-results` 没有 CSS class 定义**，仅依赖 inline style `display:none; flex-direction: column; height: 100%;`，缺少 `flex: 1` 和 `overflow: hidden`

   - `_find_source_fields` 返回前增加 bare 去重（`seen_bare` set）

2. **`api_server.py`**：
   - 新增模块级 `_bare_table()` 函数
   - `_chains_to_caliber_result` 增加口径链语义去重：chains 的 `(bare_target_table.target_column)` 签名相同的合并

### Bug2：口径详情为空（sql_block/where/join/op_type 全为空）

#### 根因
1. **`tgt_node.procedure` 为空**（BFS 节点不一定有 procedure 信息）→ 找不到存储过程 → 无法提取 sql_block
2. **`_extract_sql_block_for_mapping` 仅用 short-name 精确匹配**，无法匹配 schema 变体
3. **`_proc_target_idx` 的 key 带有 schema 前缀**（如 `RRP_EAST.EAST5_201_GRJCXXB`），而 BFS 节点可能只有 `EAST5_201_GRJCXXB`

#### 修复内容

1. **`api_server.py` `_extract_sql_block_for_mapping()`** — 3 策略匹配：
   - Strategy 1: 精确 short-name 匹配（原始逻辑）
   - Strategy 2: bare-table 归一化匹配（处理 schema 变体）
   - Strategy 3: 兜底取第一个 INSERT/MERGE 操作

2. **`api_server.py` `_chains_to_caliber_result()`** — 多策略过程查找：
   - Strategy 1: `tgt_node.procedure` 直接查找
   - Strategy 1b: fallback 到 `src_node.procedure`（修复 root 节点 procedure 为空的问题）
   - Strategy 2: `_proc_target_idx` 按 normalized/bare/bare-key-scan 查找

3. **验证**：查询 `EAST5_201_GRJCXXB.KHXM`：
   - Step 1-4 全部有 `procedure`、`op_type=INSERT_VALUES`、`sql_len>0`、`where>0` ✅
   - 之前 Step 4 的 `tgt_node.procedure` 为空问题通过 Strategy 1b（src_node.procedure fallback）解决 ✅

## 修复：详情抽屉三个显示问题（2026-05-15 16:38）

### 问题1：步骤独有WHERE/JOIN与下方WHERE/JOIN重复显示
- **根因**：前端渲染逻辑错误，当有 `step_isolated_where` 时仍显示 `where_conditions`，导致内容重复
- **修复**：`caliber-tab.js` 增加 `hasIsolatedWhere/hasIsolatedJoin` 判断，只有既无步骤独有又无累积条件时才显示 `where_conditions/join_conditions`

### 问题2：ORDER BY 内容过长
- **根因**：正则 `_ORDER_BY_PATTERN` 使用 `re.DOTALL`，匹配从 ORDER BY 到 UNION/分号/文件末尾的全部内容（可能跨多行）
- **修复**：`core/caliber_extractor.py` 改为 `[^\n;]+?`（不跨行），去掉 `re.DOTALL`，提前终止符改为 `\s*$|\s*;|\s+UNION`

### 问题3：CTE 缺少中文说明
- **根因**：前端只显示"CTE 定义"，未解释含义，用户困惑
- **修复**：`caliber-tab.js` 标题改为"CTE 定义 (Common Table Expression - 临时结果集)"
