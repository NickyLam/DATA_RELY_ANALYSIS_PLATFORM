## 展示层血缘查询 Bug 修复 ✅

### 问题描述
用户通过截图反馈展示层血缘查询的三个 Bug：
1. **Schema 自动匹配错误**：输入表名 `EAST5_201_GRJCXXB`、字段 `KHXM` 做展示层查询时，系统自动匹配了 `RRP_MDL` 用户，实际应该是 `RRP_EAST` 用户。
2. **错误用户名+表名仍能查询**：即使用户输入了错误的 schema（如 `RRP_MDL.EAST5_201_GRJCXXB`），系统仍然返回了数据，实际不应该。
3. **重复分支血缘**：`EAST5_201_GRJCXXB.KHXM` 向上追溯时出现了两条分支，其中 `ICL.CMM_INDV_CUST_BASIC_INFO` 那条是错的，应该只有一条正确分支。

### 修复方案

#### Bug 1：短名优先匹配 RRP_EAST
- 修改文件：`app/services/table_lineage_tracer.py` → `resolve_table_name()`
- 重构解析顺序：
  1. 优先从实际表定义中查找（全名或短名）
  2. 显式 schema 严格校验
  3. 短名模糊匹配时，EAST5_ 表优先匹配 RRP_EAST
  4. Schema 优先级排序：RRP_EAST > RRP_MDL > ICL
- 根因：原逻辑使用 `min(candidates, key=len)` 选择候选，`RRP_MDL`（7字符）优先于 `RRP_EAST`（8字符）

#### Bug 2：显式 schema 业务规则校验
- 修改文件：`app/services/lineage_service.py` → `query_lineage()`
- 增加业务规则校验：EAST5_ 表必须属于 RRP_EAST 用户
- 根因：`RRP_MDL.EAST5_201_GRJCXXB` 和 `RRP_EAST.EAST5_201_GRJCXXB` 都在实际表定义中存在，仅靠表定义校验无法拒绝

#### Bug 3：BFS 层级兼容性过滤
- 修改文件：`core/lineage_tracer.py` → `_bfs_trace()`
- 新增层级兼容性检查 `_is_layer_compatible()`：
  - EAST 表不应直接追溯到 ODS/DIIS 层
  - ICL schema 的表作为 EAST 表的来源通常不正确
  - OTHER 层需进一步检查（ICL_ 前缀）
- 根因：BFS 追溯未做层级过滤，EAST 表被追溯到 ODS/ICL 层

### 端对端验证结果
- Bug 1：✅ 修复成功 — `EAST5_201_GRJCXXB.KHXM` 使用 `RRP_EAST.EAST5_201_GRJCXXB`
- Bug 2：✅ 修复成功 — `RRP_MDL.EAST5_201_GRJCXXB.KHXM` 被拒绝，无数据返回
- Bug 3：✅ 修复成功 — 未发现 ICL 节点，只有正确分支
- 验证正确 schema `RRP_EAST.EAST5_201_GRJCXXB.KHXM` 能正常返回 3 节点、3 边数据 ✅
