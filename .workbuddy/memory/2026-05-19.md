## 展示层查询三 Bug 修复

### Bug 1: Schema 自动匹配错误
**问题**: 输入 `EAST5_201_GRJCXXB` 时，`resolve_table_name()` 通过 `min(candidates, key=len)` 返回 `RRP_MDL.EAST5_201_GRJCXXB`（前缀更短），而非正确的 `RRP_EAST.EAST5_201_GRJCXXB`。

**根因**: `data["tables"]` 的检查在邻接表候选返回之后，导致实际表定义未被优先使用。

**修复**: 重构 `resolve_table_name()` 解析顺序：
1. 优先从 `data["tables"]` 实际表定义中精确匹配
2. 显式 schema 前缀严格校验（未找到则返回原输入）
3. 短名解析时优先选择真实存在的表
4. EAST5_ 表优先匹配 RRP_EAST schema（业务规则）
5. 多候选时按 schema 优先级排序（RRP_EAST > RRP_MDL > ICL）

**文件**: `app/services/table_lineage_tracer.py`

### Bug 2: 错误用户名+表名仍能查询
**问题**: 输入 `RRP_MDL.EAST5_201_GRJCXXB`（错误 schema）仍能返回数据。

**根因**: 邻接表包含派生关系，`RRP_MDL.EAST5_201_GRJCXXB` 在邻接表中被直接接受，未校验是否为真实表。

**修复**: 同 Bug 1 的修复——显式 schema 前缀的输入若未在 `data["tables"]` 中找到精确匹配，直接返回原输入，后续字段追溯自然失败。

**文件**: `app/services/table_lineage_tracer.py`

### Bug 3: EAST5_201_GRJCXXB.KHXM 出现两条分支（含错误 ICL 分支）
**问题**: 字段级追溯出现 `ICL.CMM_INDV_CUST_BASIC_INFO` 分支，用户确认该分支错误。

**根因**: `_infer_source_table_from_lineage()` 在 `source_table` 为空时从 `table_lineages` 推断来源，未做层级兼容性校验，导致 EAST 表被追溯到 ODS/ICL 层。

**修复**: 在 `_infer_source_table_from_lineage()` 中增加层级兼容性过滤：
- EAST 表（层级5）不应直接追溯到 ODS/DIIS（层级0-1）
- ICL schema 的表作为 EAST 表来源视为不兼容
- ICL_ 前缀的 bare 表同样过滤

**文件**: `core/lineage_tracer.py`

## Bug 4: 口径Tab链路与展示层不一致

### 根因分析
三层问题叠加导致口径链路只有1步深度（展示层有7节点6边）：

**根因1：O_IML_ 层级误判**（`core/layer_detector.py`）
- `O_IML_AST_*` 被 `O_IML_` 前缀匹配为 ODS 层
- 实际上 `O_IML_*` 是中间层(IML)操作表，上游还有 `V_AST_*` 视图
- ODS 截断导致 BFS 在 `O_IML_AST_*` 就停止了，无法追溯 `V_AST_*`

**根因2：_paths_to_chains 记录查找方向错误**（`core/caliber_tracer.py`）
- 旧逻辑对每个 BFS 节点查 `_target_idx[node.key]`（node 作为 target 的记录）
- 但路径中 node[i] 是 node[i-1] 的上游（source），应该查 `prev_node` 作为 target、`node` 作为 source 的记录
- 导致步骤方向不匹配：查出的是"谁写入了当前节点"而非"当前节点写入了谁"

### 修复内容

**修复1：层级检测器**（`core/layer_detector.py`）
- `O_ICL_*` → ODS（核心账务系统源数据，保持不变）
- `O_IML_*` → BASE（中间层操作表，有上游视图）
- `O_IOL_*` → BASE（同理待定，暂归 BASE）
- `O_FDW_*` / `O_RDW_*` → ODS（外部数据源，保持不变）

**修复2：_paths_to_chains 重写**（`core/caliber_tracer.py`）
- 核心改动：对路径中 node[i]，查找 prev_node (path[i-1]) 作为 target 的记录中，source 匹配 node[i] 的那条
- 新增 schema 变体匹配回退：精确匹配失败后尝试 RRP_MDL/RRP_EAST/短名变体
- 回退步骤构建也统一使用 prev_node→node 方向

## 启动性能优化：缓存优先 + pickle 加速

### 问题根因
服务每次启动都全量重新解析 3041 张表 + 1754 个存储过程（耗时 6-10 分钟），即使 `lineage_data.json` 缓存文件已存在也不会复用。

更严重的是，即使有缓存加载路径，`DataRepository.load()` 仍会重复读取 2.5GB JSON 文件，导致缓存加载也需 2-3 分钟。

### 修改文件
1. **`app/services/parser_service.py`** — 核心修改
   - 新增 `load_from_cache()` 方法：优先加载 pickle（~8s），回退 JSON（~117s）
   - 新增 `_populate_result_from_data()`：从内存数据填充 ParseResult，**跳过** `repository.load()` 避免重复读 JSON
   - 新增 `_save_pickle_cache()`：保存 pickle 格式加速下次启动
   - 修改 `parse_existing_data(force=False)`：非强制时先尝试缓存
   - 修改 `get_lineage_tracer()`：支持从缓存数据重建 dataclass 对象（策略2）
   - 修改 `_save_result_to_cache()`：同时保存 pickle + JSON 双格式

2. **`app/api/system.py`** — 新增端点
   - `POST /api/system/reparse`：强制重新全量解析（清除缓存+重解析+重建索引）

3. **`app/main.py`** — lifespan 修改
   - 启动日志显示"缓存优先"，区分缓存加载 vs 全量解析

4. **`static/index.html`** / **`static/js/app.js`** — 前端
   - 顶部导航新增"🔄 重新解析"按钮
   - `forceReparse()` 函数调用 `/api/system/reparse`

### 性能对比
| 方式 | 启动耗时 | 说明 |
|------|---------|------|
| 全量解析 | 6-10 分钟 | 每次启动解析 .tab/.prc |
| JSON 缓存 | ~2 分钟 | json.load 2.5GB 慢 + DataRepository 重复读 |
| pickle 缓存 | **~10 秒** | pickle.load 8s + 索引构建 2s |

### 缓存策略
- 启动：优先 pickle → 回退 JSON（并自动生成 pickle）
- 全量解析后：同时保存 pickle + JSON（JSON 作为人可读备份）
- 手动重解析：前端按钮触发 `/api/system/reparse`，清除旧缓存后全量重解析
