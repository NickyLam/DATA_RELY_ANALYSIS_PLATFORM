# Proposal: 展示层级联查询向导（系统→用户→表→字段）

## Why

当前展示层的查询流程是"自由输入"模式：用户直接在表名输入框中搜索表名，再输入/选择字段名，最后点击查询。这种模式存在以下痛点：

1. **认知负担高**：系统有 3041 张表，用户往往不知道目标表属于哪个系统、哪个 schema，需要大量试错搜索
2. **搜索结果噪音大**：模糊搜索返回所有数据源的表，同名表（如不同 schema 下的表）容易混淆
3. **缺乏上下文引导**：用户不清楚系统中有哪些"用户"（schema/owner），无法按业务领域缩小范围
4. **字段选择盲目**：部分表有上百个字段，直接输入或浏览效率低

## What Changes

将展示层的查询入口从"自由搜索"改为"级联选择向导"：

```
[选系统] → [选用户/Schema] → [选表] → [选字段] → 输出血缘
```

### 新增前端组件
- 4 级级联下拉选择器（系统→用户→表→字段），每级选择后自动加载下一级选项
- 保留原有的自由搜索输入框作为"高级搜索"入口，满足高级用户快速查询需求

### 新增后端 API
- `GET /api/v1/systems` — 返回数据源列表（rrp, edw, mcs, fdm, tdh, gbase）
- `GET /api/v1/systems/{system}/schemas` — 返回指定系统下的 schema/用户列表
- `GET /api/v1/systems/{system}/schemas/{schema}/tables` — 返回指定 schema 下的表列表
- 现有 `GET /api/v1/tables/{table}/fields` 可复用

### 交互流程
1. 用户进入展示层，看到级联向导（默认）和高级搜索（可切换）
2. 选择系统 → 自动加载该系统下的 schema 列表
3. 选择 schema → 自动加载该 schema 下的表列表（带层级/字段数标签）
4. 选择表 → 自动加载字段列表
5. 选择字段 → 点击查询，输出血缘图谱

## Impact

### 影响的规格
- 展示层搜索面板 (`search-panel.js`) — 需新增级联组件，保留原有搜索逻辑
- 展示层 Tab 入口 (`display-tab.js`) — 需适配新的查询入口
- 后端血缘 API (`app/api/lineage.py`) — 新增 3 个端点
- 后端服务层 (`app/services/lineage_service.py`) — 新增 schema 分组、系统列表查询方法
- 前端 HTML 模板 — 新增级联选择器 DOM 结构

### 向后兼容
- 原有的自由搜索入口保留为"高级搜索"模式
- 所有现有 API 端点不变，新增端点为增量
- 前端布局变化：搜索区域从单行输入改为级联区域 + 高级搜索折叠区

### 风险评估
- **技术可行性：✅ 高** — 后端已有 `datasource_configs` 和 `search_tables` 基础，仅需新增分组查询
- **数据完整性：⚠️ 中** — 部分表可能无法归属到特定 schema（如临时表、配置表），需有"未分类"兜底
- **用户体验：✅ 高** — 级联向导大幅降低新用户认知负担
- **性能影响：✅ 低** — 级联查询每级数据量有限（系统 <10，schema <50，表 <500），可缓存

## 可行性评估

### 可行性：✅ 高度可行

| 维度 | 评估 | 说明 |
|------|------|------|
| 后端支撑 | ✅ | `DataSourceConfig` 已有系统/目录映射；`LineageService.search_tables` 支持 schema 过滤；仅需包装分组查询 |
| 前端实现 | ✅ | 级联下拉是常见 UI 模式；现有 `search-panel.js` 可扩展；D3 不受影响 |
| 数据质量 | ⚠️ | 99% 的表可归属到 schema；少量 ICL.V_* 视图需归入"未分类"；`_bare_table()` 已处理变体 |
| 性能 | ✅ | 系统/schema 列表可启动时缓存；表列表按 schema 过滤后数据量小；字段列表已有懒加载 |
| 向后兼容 | ✅ | 纯增量改动；现有搜索逻辑完整保留 |
| 工作量 | ✅ | 估计 3-5 个工作日：后端 2 个 API + 前端级联组件 + 适配 |

### 关键约束
1. **"用户"的概念定义**：在 Oracle 世界中对应 schema（如 RRP_MDL, RRP_EAST, ICL），在数仓世界中对应数据库名（如 dw, ods, dws, ads）。需统一为"命名空间"概念
2. **跨系统表**：某些血缘关系跨越系统边界（Oracle 表 → 数仓表），级联选择不应限制跨系统查询能力
3. **快速查询路径**：高级用户（如开发调试）仍需"一键搜索"能力，不能被向导拖慢
