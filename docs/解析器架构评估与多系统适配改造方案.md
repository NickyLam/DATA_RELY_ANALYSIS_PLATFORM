# 解析器架构评估与多系统适配改造方案

> **模型信息**: SOLO (Trae), 上下文窗口 ~200K tokens
>
> **版本**: v1.0 | **日期**: 2026-05-21

---

## 一、评估结论

### 1.1 核心判断：需要优化改造

当前系统在解析器架构上存在 **5 个结构性问题**，若不改造，每新增一个业务系统都需要大量硬编码修改，无法实现"同一厂商复用解析脚本、不同系统统一血缘输出"的目标。

| 问题 | 严重度 | 影响 |
|------|--------|------|
| 解析器双路径并存，Oracle 绕过 Registry | 高 | 新系统无法走统一路由 |
| 层级检测硬编码 RRP 规则 | 高 | EDW/MCS/FDM 无法正确分层 |
| 表名解析硬编码 RRP Schema | 高 | 跨系统表名无法归一化 |
| 指标体系与主血缘体系割裂 | 中 | 指标血缘无法融入统一图谱 |
| 前后端层级配置不一致 | 中 | 展示层配色错误 |

### 1.2 可行性判断

| 维度 | 评估 | 说明 |
|------|------|------|
| 同厂商复用解析脚本 | **可行** | WarehouseSQLParser 已具备两阶段解析能力，可复用于所有 SQL 类系统 |
| 不同脚本输出统一血缘结构 | **可行** | ParseOutput + ParseResult 已定义统一中间格式，需统一所有解析器走此路径 |
| 展示层正常显示多系统结果 | **需改造** | 层级检测和表名解析需可配置化，前端层级配色需动态加载 |
| 指标/口径跨系统适用 | **部分可行** | 口径功能通用性高可跨系统；指标功能与银行监管体系强绑定，需按系统定制 |

## 二、当前架构问题详细分析

### 2.1 解析器双路径并存

```
ParserService.parse_existing_data()
  │
  ├─ Oracle 数据源 ──→ 旧路径（直接调用 OracleTableParser / EnhancedProcedureParser）
  │                    绕过 ParserRegistry
  │
  └─ 其他数据源 ─────→ 新路径（Registry.parse_directory() → ParseOutput → _merge_output_to_result）
```

**问题**：Oracle 解析器不实现 `FileParser` Protocol，通过 Adapter 桥接但 ParserService 仍直接调用旧方法。新增系统时必须修改 ParserService 添加新的直接调用逻辑。

### 2.2 层级检测硬编码

[layer_detector.py](file:///Users/linmaogui/VSCodeProjects/VSCodeProjects/LLM/Trae%20SOLO/DATA_RELY_ANALYSIS_SYS/core/layer_detector.py) 中 `detect_layer()` 函数硬编码了 RRP 系统的表名前缀规则：

| 前缀 | 层级 | 适用系统 |
|------|------|---------|
| `O_ICL_*` | ODS | 仅 RRP |
| `O_IML_*/O_IOL_*` | BASE | 仅 RRP |
| `M_*` | MDL | 仅 RRP |
| `B_*` | BASE | 仅 RRP |
| `A_*/S_*` | APP | 仅 RRP |
| `EAST*` | EAST | 仅 RRP |

EDW 的 `icl/idl/iml/iol` 分层、MCS 的 `idl/itl` 分层、FDM 的指标分层 **完全无法被检测**。

### 2.3 表名解析硬编码

[table_name_resolver.py](file:///Users/linmaogui/VSCodeProjects/VSCodeProjects/LLM/Trae%20SOLO/DATA_RELY_ANALYSIS_SYS/core/table_name_resolver.py) 中：

- `KNOWN_SCHEMAS = ["ICL", "IML", "IOL", "RRP_EAST", "RRP_MDL"]` — 仅 RRP
- `default_schema = "RRP_MDL"` — 无 schema 的裸表名默认归入 RRP_MDL
- `ICL.V_* → ICL_*` 同义词映射 — 仅 RRP

### 2.4 指标体系与主体系割裂

```
主血缘体系: SQL文件 → Parser → ParseOutput → ParseResult → lineage_data.json → DataRepository → LineageService/CaliberService
指标体系:   Excel+.proc → IndicatorConfigParser → IndicatorConfigResult → IndicatorService (独立内存)
```

两套体系通过 `IndicatorService.bridge_to_field_lineage()` 单向桥接，指标血缘无法在统一图谱中展示。

### 2.5 前后端层级配置不一致

| 位置 | 层级名 |
|------|--------|
| 后端 `layer_detector.py` | `base`, `app` |
| 前端 `caliber-tab.js` | `dws`, `ads` |

基础层和应用层节点在口径 Pipeline 视图中无法正确着色。

## 三、目标架构设计

### 3.1 核心设计原则

| 原则 | 说明 |
|------|------|
| **解析器按厂商/技术栈复用** | 同一厂商实施的系统复用同一套解析脚本 |
| **统一输出格式** | 所有解析器输出相同的 ParseOutput → ParseResult 结构 |
| **层级检测可配置** | 每个系统通过 manifest.yml 定义自己的层级规则 |
| **表名解析可配置** | 每个系统定义自己的 schema 列表和默认 schema |
| **展示层动态适配** | 前端从后端 API 获取层级配置，动态渲染 |

### 3.2 解析器分类与复用矩阵

根据当前项目的数据源特征，解析器按 **技术栈/厂商** 分类复用。复用判断必须基于**实际数据内容**，而非假设：

| 解析器族 | 适用系统 | 文件类型 | 复用判断依据 |
|----------|---------|---------|------------|
| **OracleParser** | RRP (监管报送) | `.tab`, `.prc` | ✅ 已确认：Oracle 存储过程特有，独占 |
| **WarehouseParser** | EDW (数仓), MCS (管架) | `.sql`, `.ctl` | ✅ 已确认：2 系统正在复用，按 manifest 配置分层 |
| **IndicatorParser** | FDM (财务集市), RRP (监管指标) | `.xlsx`, `.proc` | ✅ 已确认：按系统定制 Excel 模板映射 |
| **HiveParser** | TDH (星环) | `.hql`, `.sql` | ✅ 已确认：基于 WarehouseParser 扩展 Hive 语法 |
| **GBaseParser** | GBase | `.sql` | ✅ 已确认：基于 WarehouseParser 扩展 GBase 语法 |
| **待定** | PAM, CCR, ICR, BRT | 未知 | ❌ 无法判断：无实际数据，技术栈未确定 |

**关键洞察**：EDW 和 MCS 可以确认复用 WarehouseParser（已有实际数据且正在使用）。PAM/CCR/ICR/BRT 目前仅为预留目录，**无实际数据和技术栈信息，无法判断能否复用现有解析器**。它们的解析器选择取决于以下因素：

| 待确定因素 | 影响 |
|-----------|------|
| 数据库类型（Oracle/Hive/GBase/其他） | 决定 SQL 方言和语法解析规则 |
| 脚本格式（.sql/.prc/.hql/.xlsx/配置文件） | 决定使用哪个解析器族 |
| 实施厂商 | 同厂商通常遵循相同脚本规范，可复用 |
| 数据加工方式（存储过程/ETL工具/配置驱动） | 决定解析策略（DDL+DML 两阶段 vs 配置解析） |
| 分层架构（是否使用 icl/idl/iml/iol 等分层） | 决定层级检测规则配置 |

**接入策略**：当 PAM/CCR/ICR/BRT 提供实际数据后，按以下决策树选择解析器：

```
新系统数据到达
  │
  ├─ 脚本为 .sql 文件 + DDL/DML 分层目录结构？
  │    └─ 是 → 复用 WarehouseParser，配置 manifest.yml 层级规则
  │
  ├─ 脚本为 Oracle .tab/.prc 文件？
  │    └─ 是 → 复用 OracleParser，配置 manifest.yml schema 映射
  │
  ├─ 数据为 Excel 配置 + .proc 存储过程？
  │    └─ 是 → 复用 IndicatorParser，定制 Excel 模板映射
  │
  ├─ 脚本为 .hql Hive 语法？
  │    └─ 是 → 复用 HiveParser（WarehouseParser 扩展）
  │
  └─ 全新格式/语法？
       └─ 实现新 Parser，实现 FileParser Protocol，注册到 Registry
```

### 4.2 改造二：LayerDetector 可配置化

**目标**：将硬编码的层级检测规则改为从 manifest.yml 动态加载。

```python
# core/layer_detector.py (改造后)

@dataclass
class LayerRule:
    pattern: str
    layer: str
    label: str

@dataclass
class LayerConfig:
    rules: list[LayerRule]
    default_schema: str
    known_schemas: list[str]
    synonym_rules: list[dict]
    layer_order: list[str]
    layer_colors: dict[str, str]

class LayerDetector:
    def __init__(self, layer_configs: dict[str, LayerConfig]):
        self._configs = layer_configs
        self._compiled_rules: dict[str, list[tuple[re.Pattern, str, str]]] = {}
        for system, config in layer_configs.items():
            self._compiled_rules[system] = [
                (re.compile(r.pattern, re.IGNORECASE), r.layer, r.label)
                for r in config.rules
            ]

    def detect_layer(self, table_name: str, system: str = "rrp") -> str:
        for pattern, layer, _ in self._compiled_rules.get(system, []):
            if pattern.search(table_name):
                return layer
        return "other"

    def get_layer_config(self, system: str) -> LayerConfig:
        return self._configs.get(system, self._configs.get("rrp"))

    @classmethod
    def from_manifests(cls, source_data_dir: Path) -> "LayerDetector":
        configs: dict[str, LayerConfig] = {}
        for system_dir in sorted(source_data_dir.iterdir()):
            manifest_path = system_dir / "manifest.yml"
            if manifest_path.exists():
                data = yaml.safe_load(manifest_path.read_text(encoding="utf-8"))
                lr = data.get("layer_rules", {})
                configs[data["system"]] = LayerConfig(
                    rules=[LayerRule(**r) for r in lr.get("rules", [])],
                    default_schema=lr.get("default_schema", "unknown"),
                    known_schemas=lr.get("known_schemas", []),
                    synonym_rules=lr.get("synonym_rules", []),
                    layer_order=lr.get("layer_order", []),
                    layer_colors=lr.get("layer_colors", {}),
                )
        return cls(configs)
```

### 4.3 改造三：TableNameResolver 可配置化

```python
            match = re.match(rule["pattern"], table_name, re.IGNORECASE)
            if match:
                return match.expand(rule["target"])
        return table_name
```

### 4.4 改造四：ParserService 统一路由

**目标**：消除双路径，所有数据源统一走 Registry 路由。

```python
# app/services/parser_service.py (改造后核心逻辑)

class ParserService:
    def __init__(self, source_data_dir: str, output_dir: str):
        self._registry = ParserRegistry()
        self._layer_detector = LayerDetector.from_manifests(Path(source_data_dir))
        self._table_name_resolver = TableNameResolver(self._layer_detector)

        # 从全局 manifest.yml 加载所有数据源配置
        self._datasource_configs = self._load_manifests(Path(source_data_dir))

        # 注册解析器（按 parser 类型）
        self._register_parsers()

    def _register_parsers(self):
        oracle_parser = OracleParser(self._layer_detector, self._table_name_resolver)
        warehouse_parser = WarehouseParser(self._layer_detector, self._table_name_resolver)

        self._registry.register(".tab", oracle_parser)
        self._registry.register(".prc", oracle_parser)
        self._registry.register(".sql", warehouse_parser)
        self._registry.register(".ctl", warehouse_parser)

    def parse_all(self) -> ParseResult:
        result = ParseResult()
        for ds_config in self._datasource_configs:
            if not ds_config.enabled:
                continue
            system_dir = Path(self._source_data_dir) / ds_config.path
            output = self._registry.parse_directory(system_dir, ds_config.system)
            result.merge(output)
        return result
```

### 4.5 改造五：WarehouseParser 支持多系统配置注入

**目标**：WarehouseParser 接收系统配置，按不同系统的层级规则和 schema 映射解析。

```python
# core/warehouse/warehouse_parser.py (改造后)

class WarehouseParser:
    def __init__(
        self,
        layer_detector: LayerDetector,
        table_name_resolver: TableNameResolver,
    ):
        self._layer_detector = layer_detector
        self._table_name_resolver = table_name_resolver

    def parse_directory(self, dir_path: Path, system: str = "edw") -> ParseOutput:
        config = self._layer_detector.get_layer_config(system)

        # Phase 1: DDL
        ddl_parser = DDLParser(
            schema_resolver=SchemaResolver(config.known_schemas),
            layer_detector=self._layer_detector,
            system=system,
        )
        tables_dict = ddl_parser.parse_directory(dir_path / "ddl")

        # Phase 2: DML
        dml_parser = DMLParser(
            tables_dict=tables_dict,
            schema_resolver=SchemaResolver(config.known_schemas),
            layer_detector=self._layer_detector,
            system=system,
        )
        dml_output = dml_parser.parse_directory(dir_path / "dml")

        # Phase 3: CTL (如果存在)
        ctl_output = ParseOutput()
        ctl_dir = dir_path / "ext"
        if ctl_dir.exists():
            ctl_parser = CTLParser(schema_resolver=SchemaResolver(config.known_schemas))
            ctl_output = ctl_parser.parse_directory(ctl_dir)

        # 合并
        ddl_output = self._tables_to_output(tables_dict, system)
        return ddl_output.merge(dml_output).merge(ctl_output)
```

### 4.6 改造六：IndicatorParser 适配器化

**目标**：为 IndicatorConfigParser 创建适配器，使其融入统一 ParseOutput 体系。

```python
# core/adapters/indicator_adapter.py (新增)

class IndicatorAdapter:
    """将 IndicatorConfigParser 的输出转换为 ParseOutput 格式"""

    def __init__(self, layer_detector: LayerDetector, system: str = "fdm"):
        self._layer_detector = layer_detector
        self._system = system

    def parse_directory(self, dir_path: Path) -> ParseOutput:
        parser = IndicatorConfigParser(data_path=str(dir_path))
        result = parser.parse_all()

        output = ParseOutput()

        # 将指标依赖关系转换为 TableLineage
        for rel in result.relations:
            output.table_lineages.append({
                "source_table": rel.source_table,
                "target_table": rel.target_table,
                "procedure": rel.procedure_name,
                "system": self._system,
            })

        # 将指标字段映射转换为 FieldMapping
        for calc in result.base_calcs:
            for mapping in calc.field_mappings:
                output.field_mappings.append({
                    "source_table": mapping.source_table,
                    "source_column": mapping.source_column,
                    "target_table": calc.trg_table_name,
                    "target_column": mapping.target_column,
                    "transform_logic": mapping.expression,
                    "system": self._system,
                })

        # 将指标表注册到 tables
        for table_name in set(
            [c.trg_table_name for c in result.base_calcs]
            + [c.src_table_name for c in result.base_calcs]
        ):
            layer = self._layer_detector.detect_layer(table_name, self._system)
            output.tables.append({
                "full_name": table_name,
                "table_name": table_name,
                "schema": self._layer_detector.get_layer_config(self._system).default_schema,
                "layer": layer,
                "system": self._system,
            })

        return output
```

## 六、改造后新增系统接入流程

改造完成后，新增一个业务系统（如 PAM 绩效考核系统）只需：

```bash
mkdir -p SOURCE_DATA/PAM/{ddl,dml,config}
```

### 步骤 2：放入数据文件

### 步骤 3：注册到全局 manifest

在 `SOURCE_DATA/manifest.yml` 的 `sources` 中添加：

```yaml
  - name: pam
    display_name: "绩效考核系统"
    path: PAM
    enabled: true
    parser: warehouse
    file_extensions: [".sql"]
```

### 完成！

无需修改任何 Python 代码或前端代码。PAM 系统的数据将：
- 自动被 WarehouseParser 解析（复用同一套脚本）
- 自动按 manifest.yml 定义的层级规则分层
- 自动在血缘图、口径 Pipeline、指标查询中正确展示
