一、评估结论

⚠️ 核心判断：需要优化改造

当前系统在解析器架构上存在 5 个结构性问题，若不改造，每新增一个业务系统都需要大量硬编码修改，无法实现"同一厂商复用解析脚本、不同系统统一血缘输出"的目标。

问题清单

问题严重度影响
解析器双路径并存，Oracle 绕过 Registry高新系统无法走统一路由
层级检测硬编码 RRP 规则高EDW/MCS/FDM 无法正确分层
表名解析硬编码 RRP Schema高跨系统表名无法归一化
指标体系与主血缘体系割裂中指标血缘无法融入统一图谱
前后端层级配置不一致中展示层配色错误

可行性判断

✅ 同厂商复用解析脚本

WarehouseParser 已具备两阶段解析能力，可复用于所有 SQL 类系统

✅ 统一血缘输出结构

ParseOutput + ParseResult 已定义统一中间格式，需统一所有解析器走此路径

🔧 展示层多系统适配

层级检测和表名解析需可配置化，前端层级配色需动态加载

🔧 指标/口径跨系统

口径功能通用性高可跨系统；指标功能与银行监管体系强绑定，需按系统定制

二、当前架构问题详细分析

2.1 解析器双路径并存

ParserService.parse_existing_data()
  │
  ├─ Oracle 数据源 ──→ 旧路径（直接调用 OracleTableParser / EnhancedProcedureParser）
  │                    绕过 ParserRegistry
  │
  └─ 其他数据源 ─────→ 新路径（Registry.parse_directory() → ParseOutput → _merge_output_to_result）

Oracle 解析器不实现 FileParser Protocol，通过 Adapter 桥接但 ParserService 仍直接调用旧方法。新增系统时必须修改 ParserService 添加新的直接调用逻辑。

2.2 层级检测硬编码

layer_detector.py 中 detect_layer() 函数硬编码了 RRP 系统的表名前缀规则：

前缀层级适用系统
O_ICL_*ODS仅 RRP
O_IML_*/O_IOL_*BASE仅 RRP
M_*MDL仅 RRP
B_*BASE仅 RRP
A_*/S_*APP仅 RRP
EAST*EAST仅 RRP

EDW 的 icl/idl/iml/iol 分层、MCS 的 idl/itl 分层、FDM 的指标分层 完全无法被检测。

2.3 表名解析硬编码

table_name_resolver.py 中：

  • KNOWN_SCHEMAS = ["ICL", "IML", "IOL", "RRP_EAST", "RRP_MDL"] — 仅 RRP
  • default_schema = "RRP_MDL" — 无 schema 的裸表名默认归入 RRP_MDL
  • ICL.V_* → ICL_* 同义词映射 — 仅 RRP

2.4 指标体系与主体系割裂

主血缘体系: SQL文件 → Parser → ParseOutput → ParseResult → lineage_data.json → DataRepository → LineageService/CaliberService
指标体系:   Excel+.proc → IndicatorConfigParser → IndicatorConfigResult → IndicatorService (独立内存)

两套体系通过 IndicatorService.bridge_to_field_lineage() 单向桥接，指标血缘无法在统一图谱中展示。

2.5 前后端层级配置不一致 BUG

位置层级名
后端 layer_detector.pybase, app
前端 caliber-tab.jsdws, ads

基础层和应用层节点在口径 Pipeline 视图中无法正确着色。

三、目标架构设计

3.1 核心设计原则

解析器按厂商复用

同一厂商实施的系统复用同一套解析脚本

统一输出格式

所有解析器输出相同的 ParseOutput → ParseResult 结构

层级检测可配置

每个系统通过 manifest.yml 定义自己的层级规则

展示层动态适配

前端从后端 API 获取层级配置，动态渲染

3.2 解析器分类与复用矩阵

复用判断必须基于实际数据内容，而非假设：

⚠️ 严谨声明：EDW 和 MCS 可以确认复用 WarehouseParser（已有实际数据且正在使用）。PAM/CCR/ICR/BRT 目前仅为预留目录，无实际数据和技术栈信息，无法判断能否复用现有解析器。它们的解析器选择取决于数据库类型、脚本格式、实施厂商、数据加工方式等因素。

改造② LayerDetector 可配置化

@dataclass
class LayerRule:
    pattern: str
    layer: str
    label: str

@dataclass
class LayerConfig:
    rules: list[LayerRule]
    default_schema: str
    known_schemas: list[str]
    synonym_rules: list[dict]
    layer_order: list[str]
    layer_colors: dict[str, str]

class LayerDetector:
    def __init__(self, layer_configs: dict[str, LayerConfig]):
        self._configs = layer_configs
        self._compiled_rules: dict[str, list[tuple[re.Pattern, str, str]]] = {}
        for system, config in layer_configs.items():
            self._compiled_rules[system] = [
                (re.compile(r.pattern, re.IGNORECASE), r.layer, r.label)
                for r in config.rules
            ]

    def detect_layer(self, table_name: str, system: str = "rrp") -> str:
        for pattern, layer, _ in self._compiled_rules.get(system, []):
            if pattern.search(table_name):
                return layer
        return "other"

    @classmethod
    def from_manifests(cls, source_data_dir: Path) -> "LayerDetector":
        ...

改造④ ParserService 统一路由

class ParserService:
    def __init__(self, source_data_dir: str, output_dir: str):
        self._registry = ParserRegistry()
        self._layer_detector = LayerDetector.from_manifests(Path(source_data_dir))
        self._table_name_resolver = TableNameResolver(self._layer_detector)
        self._datasource_configs = self._load_manifests(Path(source_data_dir))
        self._register_parsers()

    def _register_parsers(self):
        oracle_parser = OracleParser(self._layer_detector, self._table_name_resolver)
        warehouse_parser = WarehouseParser(self._layer_detector, self._table_name_resolver)
        self._registry.register(".tab", oracle_parser)
        self._registry.register(".prc", oracle_parser)
        self._registry.register(".sql", warehouse_parser)
        self._registry.register(".ctl", warehouse_parser)

    def parse_all(self) -> ParseResult:
        result = ParseResult()
        for ds_config in self._datasource_configs:
            if not ds_config.enabled:
                continue
            system_dir = Path(self._source_data_dir) / ds_config.path
            output = self._registry.parse_directory(system_dir, ds_config.system)
            result.merge(output)
        return result

六、改造后新增系统接入流程

✅ 三步零代码接入

1
创建目录 + manifest
mkdir -p SOURCE_DATA/PAM/{ddl,dml,config}
编写 manifest.yml 定义层级规则
2
放入数据文件
将 SQL 脚本按 ddl/ + dml/ 放入对应目录
3
注册到全局 manifest
在 SOURCE_DATA/manifest.yml 的 sources 中添加一条记录

完成后，PAM 系统的数据将：

  • 自动被 WarehouseParser 解析（复用同一套脚本）
  • 自动按 manifest.yml 定义的层级规则分层
  • 自动在血缘图、口径 Pipeline、指标查询中正确展示

七、风险与回退策略

风险概率影响缓解措施
LayerDetector 重构导致 RRP 回归中高RRP 的 manifest.yml 与当前硬编码规则完全一致，先验证 RRP 无回归
WarehouseParser 多系统注入 Bug中中system 参数默认值 "rrp" 保证向后兼容
前端动态加载增加延迟低低层级配置 API 响应体 < 1KB，可缓存在 localStorage
IndicatorAdapter 转换精度不足中中保留 IndicatorService 独立路径，Adapter 仅用于统一图谱展示
