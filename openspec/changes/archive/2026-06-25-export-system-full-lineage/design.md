## Context

The application already parses all source systems into a current dataset containing `tables`, `procedures`, `table_lineages`, `field_mappings`, and metadata. Existing lineage APIs focus on interactive search and bounded upstream/downstream queries, while `/api/systems` and `/api/systems/{system_name}/tables` expose system-scoped table browsing.

Full lineage export should use the same parsed dataset and system classification rules as the existing cascade table queries. It must not require clients to issue one request per table, and it must not depend on frontend graph layout state. The user-facing artifact is an `.xlsx` workbook so analysts can inspect, filter, archive, and share the full system lineage outside the web UI.

## Goals / Non-Goals

**Goals:**
- Provide one read-only API endpoint to export the complete lineage graph for a selected system.
- Return an XLSX download that downstream users can open directly in spreadsheet tools.
- Keep system filtering consistent with existing system/table cascade behavior, including `data_source` preference and schema-based fallback.
- Include enough metadata and counts for users to verify what was exported.
- Keep workbook generation deterministic and testable by separating workbook-ready data assembly from XLSX serialization.
- Preserve existing table/field lineage query behavior.

**Non-Goals:**
- Add server-side scheduled exports or persistent export files.
- Add CSV, JSON, image, or graph-layout exports in the first implementation.
- Reparse source files during export.
- Change lineage parsing semantics or fix parser correctness issues unrelated to export.

## Decisions

1. Add `GET /api/systems/{system_name}/lineage/export`.
   - Rationale: the feature is naturally scoped below a system, and the route sits beside the existing `/api/systems/{system_name}/tables` cascade API.
   - Alternative considered: `GET /api/lineage/export?system=...`; rejected because it hides the primary resource identity in a query parameter.

2. Assemble export data in `LineageService`, not the route handler.
   - Rationale: the route should validate HTTP concerns and return a file response, while filtering and lineage graph assembly are service responsibilities.
   - Alternative considered: reuse `TableQueryService` directly from the endpoint; rejected because export must include lineage edges and field mappings, not only table lists.

3. Reuse or extract the existing system classification logic.
   - Rationale: current cascade APIs prefer explicit `data_source`, then schema-to-system mapping, with special handling for non-Oracle systems. Export must not classify tables differently from the UI table selector.
   - Implementation direction: move shared helpers such as schema-to-system construction and table-to-system matching into a small reusable helper, or call a shared service method if that keeps dependencies clean.

4. Export raw lineage semantics, not rendered graph layout.
   - Rationale: downstream consumers need durable data contracts: tables as nodes, table lineage records as edges, field mappings as mappings, and metadata/counts. Layout coordinates and D3-specific state are frontend concerns.
   - Alternative considered: export the current D3 graph shape from `query_lineage`; rejected because it is query-depth-bound and optimized for one target, not a complete system.

5. Include cross-system relationships when they touch the selected system.
   - Rationale: a complete view of a system needs edges and mappings where either source or target belongs to the system; otherwise inbound or outbound dependencies would be hidden.
   - Representation: nodes include selected-system tables and referenced external tables; each node carries `system_name` when known and `is_external` when outside the selected system.

6. Generate the workbook with `openpyxl` into an in-memory `BytesIO`.
   - Rationale: `openpyxl` is already a project dependency, supports `.xlsx`, and can be tested by loading the generated bytes back into a workbook.
   - Workbook sheets:
     - `Summary`: system name, generated time, data timestamp/generation when available, and counts.
     - `Tables`: one row per exported node, including full name, short name, layer, field count, system name, and external flag.
     - `TableLineages`: one row per table-level edge, including source, target, procedure, data source, and any available operation metadata.
     - `FieldMappings`: one row per field mapping, including source/target table and column, procedure, transformation logic, and condition metadata when present.
   - Alternative considered: return JSON and let the frontend create XLSX; rejected because it duplicates workbook logic across clients and weakens API-level verification.

## Risks / Trade-offs

- Large systems can produce large workbooks -> keep the first implementation in-memory with `BytesIO`, include counts, avoid layout expansion, and consider streaming/background export only if workbook size becomes a measured problem.
- Existing system classification is duplicated in `TableQueryService` -> mitigate by extracting shared helper logic or testing export against the same classification cases.
- Some lineage rows may lack `data_source` on source/target tables -> mitigate by resolving through table metadata and schema fallback before marking a node external.
- Field mappings may contain procedure-only or unresolved tables not present in `tables` -> keep them in the export if they touch the selected system, and include referenced table nodes with minimal metadata.
- Binary responses do not use a normal Pydantic response body -> document the XLSX media type and filename headers explicitly in FastAPI route metadata and tests.
