## Context

字段定义已经由 Oracle、数仓和 PAM DDL 解析器写入 `ColumnInfo.data_type`，并通过表数据的 `columns` 集合保存到 JSON/SQLite。类型值是完整声明，例如 `VARCHAR2(60)`、`NUMBER(18,2)` 和 `DATE`，因此解析层和存储层已经具备本次展示所需的数据。

当前缺口位于查询与展示之间：

- `LineageService._build_nodes()` 只输出表级节点；当 `include_fields=false` 时不携带任何字段元数据。
- `static/js/lineage-graph.js` 的 `_getNodeField()` 从查询目标、边或映射中选择字段名，但无法取得字段类型。
- 当前页面实际加载模块化的 `lineage-graph.js` 和 `detail-panel.js`；`display-tab.js` 中存在旧的重复实现，但未被 `static/index.html` 引入。
- 图查询默认使用 `include_fields=false`，不能通过恢复整表 `columns` 来解决，否则会扩大响应体并抵消现有懒加载设计。

## Goals / Non-Goals

**Goals:**

- 让每个有当前链路字段的节点获得准确、可直接渲染的字段名和完整类型声明。
- 在不返回整表字段列表的前提下支持 `VARCHAR2(60)`、`NUMBER(18,2)`、`DATE` 等展示。
- 对缺少 DDL、仅有表级血缘、旧响应和旧缓存形状保持兼容。
- 保持字段选择逻辑确定且可测试，避免前端和详情面板各自推断出不同字段。

**Non-Goals:**

- 不修改 DDL 解析规则，也不把 `data_type` 拆成新的持久化列。
- 不从 SQL 表达式推断未知字段类型，不进行跨节点类型兼容性校验。
- 不在首屏返回每张表的全部字段定义。
- 不清理 `display-tab.js` 的历史重复代码；该问题属于独立的前端清理范围。
- 不改变指标血缘图 `indicator-tab.js` 的节点展示。

## Decisions

### 1. 在节点上增加可选的 `field` 对象

`POST /api/lineage/query` 的每个节点按需增加：

```json
{
  "field": {
    "name": "CUSTOMER_NAME",
    "data_type": "VARCHAR2(60)"
  }
}
```

选择对象而不是新增 `field_name`、`field_type` 两个平铺属性，是为了明确这些属性属于同一个链路字段，并保留后续增加 `nullable`、`comment` 的扩展空间。该字段为可选项，不影响现有客户端。

备选方案是重新启用 `include_fields=true` 并返回完整 `columns`。该方案会显著放大多节点图的响应体，并违背当前节点详情懒加载策略，因此不采用。

### 2. 保留原始 `data_type`，不单独解析 length

后端直接返回解析器保存的完整 `data_type`。括号中的内容可能是字符长度、数值精度/小数位、时间精度或数据库特定参数，统一称为“类型声明”更准确。前端直接展示该字符串，避免用正则拆分后错误解释 `NUMBER(18,2)`。

备选方案是返回 `type_name`、`length`、`precision`、`scale`。这会引入数据库方言解析和模型迁移，超出展示需求。

### 3. 后端统一选择节点的当前链路字段

在 `_assemble_result()` 中先基于查询目标、字段边和字段映射建立 `node -> selected field` 映射，再传给 `_build_nodes()`：

1. 查询目标节点优先使用请求中的 `field`。
2. 其他节点按当前结果中字段级边的稳定顺序选择对应端点字段。
3. 边信息不足时使用 `field_mappings` 补充。
4. 只有表级关系且无法确定字段时，不生成 `field`。

字段类型从节点对应的 `table_info.columns` 中按大小写不敏感方式查找。表匹配优先完整名，再使用现有 `TableNameResolver`/短名索引兼容映射中的短表名。

备选方案是继续由 `_getNodeField()` 在浏览器中扫描边。该方案不能访问服务端列定义，而且会让图节点与详情面板重复实现选择逻辑，因此不采用。

### 4. 不新增持久化迁移或全局索引

节点数默认限制为 100，`LineageQueryIndex.table_by_full` 已能 O(1) 找到表；在命中的表内对 `columns` 做一次字段匹配，复杂度和内存开销可控。先使用查询期局部映射，只有性能测试证明列扫描成为热点时，才增加 `(table, field) -> column` 全局索引。

### 5. 前端优先使用节点元数据并保留旧逻辑降级

`lineage-graph.js` 增加单一格式化函数，优先读取 `node.field`：

- 有类型：展示字段名和完整 `data_type`。
- 无类型：只展示字段名。
- 无 `node.field`：调用现有 `_getNodeField()` 兼容旧响应。

长文本按 SVG 可用宽度截断，并在节点组或文本元素添加 `<title>` 显示完整值。点击节点时将同一个字段对象传给 `showInfoPanel()`；`detail-panel.js` 同时兼容旧的字符串参数，避免一次性破坏其他调用方。

### 6. 测试围绕契约和降级行为

- 服务层单元测试验证节点字段选择、完整/短表名匹配和原始类型保留。
- API 测试验证 `include_fields=false` 时仍返回 `node.field`，同时不返回完整 `columns`。
- 前端测试验证 `VARCHAR2(60)`、`NUMBER(18,2)`、`DATE`、空类型和旧响应的渲染结果。
- E2E 覆盖一个真实查询节点的可见字段类型和节点点击详情。

## Risks / Trade-offs

- [同一表节点涉及多个字段时只选择一个主展示字段] → 保持与当前单字段节点卡片一致，以查询目标优先并稳定选择；完整映射仍保留在边和详情中，后续可独立扩展多字段节点布局。
- [部分过程临时表没有 DDL，无法取得类型] → 返回字段名和空 `data_type`，不伪造类型，也不阻断图渲染。
- [类型文本使节点标签过长] → 使用安全截断和 SVG `title`，必要时微调节点高度/字体，但不无限扩大节点宽度。
- [查询缓存中可能存在旧节点结构] → 前端保留边推断降级；服务重启或缓存重建后自动生成新结构，无持久化数据迁移。
- [短表名在多个 schema 中重名] → 完整名优先；只有唯一或现有解析规则可确定时才使用短名匹配，否则保留字段名但不附加可能错误的类型。

## Migration Plan

1. 先增加后端节点字段组装和自动化测试，保持字段可选。
2. 更新模块化图渲染和详情面板，优先消费新结构并保留旧响应降级。
3. 更新静态资源版本参数，避免浏览器继续使用旧 JavaScript。
4. 运行聚焦测试和完整 pytest；通过本地浏览器验证节点文本、tooltip 和详情面板。
5. 回滚时可仅回退前端消费逻辑；新增的可选 API 字段不会影响旧客户端。若后端也回退，前端仍可使用旧字段名推断路径。

## Open Questions

- 当前方案延续“每个表节点显示一个主链路字段”。若产品期望同一节点同时展开所有参与映射的字段及类型，需要单独定义多字段节点的高度、折叠和边连接规则。
